# AGENTS.md

## Назначение проекта

Этот репозиторий предназначен для чат-бота, с которым можно взаимодействовать через:

- Telegram
- HTTPS-домен / web endpoint

В фоне бот использует OpenAI API и должен поддерживать сохранение памяти между сессиями.

## Ключевые требования к продукту

- Обеспечить единый пользовательский опыт для бота как через Telegram, так и через HTTPS-интерфейс.
- По возможности держать бизнес-логику независимой от транспорта, чтобы слои Telegram и HTTPS оставались тонкими.
- Использовать OpenAI API как основной LLM-бэкенд.
- Реализовать персистентную память между сессиями для пользователей/чатов там, где это требуется продуктовой логикой.
- Спроектировать систему так, чтобы поведение памяти было явным, наблюдаемым и пригодным для тестирования.

## Инженерные принципы

- Предпочитать чистое разделение между транспортным слоем, прикладной логикой, интеграцией с OpenAI, слоем памяти и хранилищем.
- Добавлять новый функционал небольшими, понятными для ревью итерациями.
- Избегать скрытого состояния и неявных побочных эффектов.
- Изолировать конфигурацию, секреты и средозависимое поведение от доменной логики.

## Политика тестирования

Весь новый функционал должен быть покрыт тестами.

Обязательные требования:

- Каждая новая фича должна сопровождаться автоматизированными тестами.
- Исправления багов по возможности должны включать regression tests.
- Критические сценарии должны иметь покрытие на уровне интеграционных тестов.
- Ключевая бизнес-логика должна быть покрыта unit-тестами.
- Поведение, связанное с памятью, должно тестироваться явно, включая персистентность и границы между сессиями.
- Потоки, критичные для логирования, должны тестироваться там, где это практично, особенно для обработки ошибок и observability-сценариев.

Нельзя считать работу завершенной или готовой к merge, если функциональность добавлена без достаточного тестового покрытия.

## Логирование и наблюдаемость

Прозрачное логирование обязательно для удобного debug и дальнейшей поддержки.

Обязательные требования:

- Логировать все важные события жизненного цикла в структурированном и читаемом виде.
- Логировать запросы и ответы с безопасным уровнем детализации, не раскрывая секреты и чувствительные пользовательские данные.
- Логировать взаимодействия с OpenAI API, хранилищем памяти, внешними транспортами, retry-механизмами и ошибками.
- По возможности использовать correlation id, чтобы одно пользовательское взаимодействие можно было проследить end-to-end.
- Логи ошибок должны содержать достаточно контекста для диагностики без необходимости воспроизводить проблему вслепую.
- Debug-логи должны помогать понять ход принятия решений, особенно вокруг загрузки/сохранения памяти и вызовов модели.

Никогда не логировать:

- API keys
- access tokens
- raw secrets
- чувствительные персональные данные, если это явно не требуется и не прошло корректную санацию

## Требования к памяти

- Память должна сохраняться между сессиями в соответствии с правилами продукта.
- Область действия памяти должна быть определена явно: на пользователя, на чат, на диалог или по другой явно зафиксированной границе.
- Чтение и запись памяти должны быть детерминированными и наблюдаемыми через логи.
- Обновления памяти должны быть намеренными, а не побочным эффектом несвязанного кода.
- Если вводятся правила summarization, truncation или retention policy, они должны быть задокументированы и покрыты тестами.

## Ожидания от агентов

При внесении изменений в этот репозиторий агенты должны:

- обновлять или добавлять тесты вместе с новой функциональностью
- добавлять или улучшать логирование, необходимое для отладки
- держать модули небольшими, а зоны ответственности понятными
- документировать все неочевидные архитектурные или операционные решения
- избегать непрозрачной "магии" в поведении системы

## Критерии готовности

Работа считается завершенной только если:

- запрошенная функциональность реализована
- для нового поведения добавлены автоматизированные тесты
- релевантные тесты проходят локально, если текущее устройство проекта это позволяет
- логирование достаточно для последующего debug
- в коде и логах не раскрываются секреты
