# MEMORY.md

## Назначение

Этот документ описывает целевую реализацию памяти для Telegram-бота.

Цели:

- сохранять память между сессиями
- держать отдельную память для каждого пользователя Telegram
- использовать только последние `N` реплик активного диалога в оперативном контексте
- суммаризировать завершенные диалоги целиком после периода неактивности
- хранить и подавать в контекст устойчивую персональную информацию пользователя
- хранить и подавать в контекст только последние `K` суммаризаций прошлых диалогов
- ограничивать общий memory-блок бюджетом `C`

## Ключевые параметры

- `N` — количество последних реплик активного диалога, которые попадают в контекст ответа
- `K` — количество последних суммаризаций завершенных диалогов, которые могут попасть в контекст
- `C` — бюджет контекста для memory-блока: `personal + K последних summaries`
- `session_timeout_seconds` — период неактивности, после которого диалог считается завершенным; по умолчанию `3600`

Все параметры должны задаваться при старте приложения.

## Общая модель памяти

Память делится на два слоя:

Ключевое инвариантное правило:

- ограничение `N` действует только на оперативный контекст активного диалога при генерации очередного ответа
- ограничение `N` не должно использоваться при суммаризации завершенного диалога
- суммаризация всегда должна строиться по всему диалогу целиком, используя всю его текстовую историю и текстовые описания вложений, но не должна включать добавленные в начале суммаризации и персональную информацию

### 1. Short-term memory

Это активный диалог пользователя.

Хранит:

- все реплики текущей активной сессии в persistent storage
- в модель при ответе отправляются только последние `N` реплик этой сессии

Назначение:

- поддерживать естественное продолжение текущего разговора
- не раздувать prompt всем историческим диалогом

### 2. Long-term memory

Это память между сессиями.

Состоит из:

- `personal memory` — устойчивые факты о пользователе
- `dialog summaries` — суммаризации прошлых завершенных диалогов

Назначение:

- переносить важный контекст между сессиями
- не хранить в prompt полную сырую историю всех диалогов

## Ключ памяти

Основной ключ памяти — `telegram_user_id`.

Это означает:

- у каждого пользователя Telegram своя отдельная память
- память не делится между разными пользователями
- все прошлые диалоги одного пользователя сводятся в общую долговременную память

## Жизненный цикл диалога

### Активная сессия

У пользователя может быть только одна активная сессия.

При новом сообщении:

1. Находим активную сессию пользователя.
2. Если активной сессии нет, создаем новую.
3. Если активная сессия есть, но с момента последней активности прошло больше `session_timeout_seconds`, текущую сессию закрываем и создаем новую.
4. Новое сообщение записываем уже в актуальную активную сессию.

### Завершение сессии

Если пользователь не писал больше часа, сессия считается завершенной.

После завершения:

- сессия переводится в состояние `closed_pending_summary`
- по ней должна пройти суммаризация
- после успешной суммаризации сессия переводится в состояние `summarized`

Важно:

- при ответе модели используется только `N` последних реплик активной сессии
- при суммаризации завершенной сессии используются все сообщения этой сессии

## Что именно суммаризируется

Суммаризация должна проходить:

- по всей текстовой части завершенного диалога
- по текстовым описаниям прикрепленных документов и файлов

Суммаризация не должна ограничиваться последними `N` репликами.

Сами вложения повторно обрабатываться не должны.

Это обязательное правило.

## Представление сообщений

Каждое сообщение должно храниться в двух видах:

### 1. Полный контент сообщения

Например:

- текст пользователя
- `input_file`
- `input_image`
- ответ ассистента

Это нужно для:

- ответа внутри текущего активного диалога
- работы мультимодального сценария в моменте

### 2. Текст для суммаризации

Это отдельное текстовое представление сообщения, пригодное для будущей суммаризации.

Примеры:

- текст пользователя:
  `Пользователь: хочу понять, что написано в документе`
- ответ ассистента:
  `Ассистент: документ описывает порядок начала работы с системой`
- пользователь прислал PDF:
  `Пользователь прикрепил PDF "Инструкция_по_началу_работы_с_ЛИИС.pdf" и задал вопрос: "о чем тут?"`
- пользователь прислал XLSX:
  `Пользователь прикрепил XLSX "report.xlsx" и попросил сделать краткую сводку по таблице`

Именно эти текстовые представления должны использоваться в суммаризации завершенной сессии.

Повторная отправка в модель самих PDF, DOCX, XLSX, изображений и других файлов для суммаризации запрещена.

## Structured output для суммаризации

Суммаризация завершенного диалога должна выполняться через structured output и возвращать два блока данных:

### 1. `personal`

Это устойчивые факты о пользователе, которые полезны в будущих диалогах.

Примеры:

- имя
- роль
- профессия
- язык общения
- устойчивые предпочтения
- рабочий контекст
- долгосрочные цели

Требования:

- извлекать только относительно стабильную информацию
- не включать ситуативные факты одного разговора, если они не выглядят устойчивыми
- не дублировать уже известные факты без необходимости

### 2. `dialog_summary`

Это ключевая информация только по конкретному завершенному диалогу.

Содержимое:

- краткая суть разговора
- важные выводы
- упомянутые документы
- принятые решения
- открытые вопросы
- полезные договоренности или задачи

Этот блок относится только к одному завершенному диалогу.

## Как формируется контекст нового ответа

При каждом новом запросе в модель контекст должен собираться так:

1. Системный промпт.
2. Блок `personal memory`.
3. Последние `K` суммаризаций прошлых завершенных диалогов.
4. Последние `N` реплик активной сессии.
5. Текущее сообщение пользователя.

## Ограничение через `C`

Бюджет `C` относится только к memory-блоку:

- `personal memory`
- последние `K` dialog summaries

Правило:

- сначала в память всегда пытаемся включить `personal memory`
- затем добавляем summaries от новых к старым
- если добавление очередной summary превышает `C`, она не включается
- в prompt попадает только тот объем memory, который помещается в `C`

Если `personal memory` сама по себе превышает `C`, нужно:

- либо обрезать/сжимать `personal memory`
- либо прогонять отдельную нормализацию personal memory

Но в любом случае итоговый memory-блок не должен превышать `C`.

## Хранилище

Для первой реализации рекомендуется SQLite.

Причины:

- быстро внедряется
- подходит для одного инстанса бота
- дает настоящую persistent memory между перезапусками
- легко тестируется

Хранилище должно быть централизованным относительно процесса бота:

- все пользователи пишутся в одну БД
- память всех пользователей хранится в одной структуре

Если в будущем потребуется несколько инстансов приложения, хранилище можно заменить на Postgres, но интерфейсы должны проектироваться так, чтобы бизнес-логика памяти от этого не зависела.

## Предлагаемые сущности хранения

### `users`

- `telegram_user_id`
- `created_at`
- `updated_at`

### `sessions`

- `id`
- `telegram_user_id`
- `status`
- `started_at`
- `last_activity_at`
- `closed_at`
- `summarized_at`

Статусы:

- `open`
- `closed_pending_summary`
- `summarized`

### `messages`

- `id`
- `session_id`
- `role`
- `created_at`
- `content_json`
- `summary_text`

### `personal_memory`

- `telegram_user_id`
- `content_json`
- `updated_at`

### `session_summaries`

- `id`
- `session_id`
- `telegram_user_id`
- `summary_json`
- `created_at`

## Работа с personal memory

`personal memory` должна храниться отдельно от суммаризаций диалогов.

После каждой суммаризации завершенной сессии:

1. Из structured output берем `personal`.
2. Мержим эти факты с уже существующей `personal memory`.
3. Обновляем централизованную запись пользователя.

Базовые правила merge:

- удалять явные дубли
- сохранять более конкретную и полезную формулировку
- не смешивать противоречивые факты без проверки
- не раздувать память мелкими ситуативными деталями

Для первой версии merge можно делать детерминированным кодом.

## Работа с dialog summaries

После суммаризации завершенной сессии:

- создается одна запись в `session_summaries`
- она относится только к конкретной завершенной сессии
- в общий prompt попадает только из числа последних `K`

Summaries должны храниться в порядке времени и легко выбираться от новых к старым.

## Где должна происходить суммаризация

Суммаризация не должна блокировать обычный ответ пользователю.

Рекомендуемый вариант:

- при каждом цикле polling дополнительно запускать проверку просроченных сессий
- если сессия неактивна больше часа, переводить ее в `closed_pending_summary`
- затем отдельно вызывать summarization pipeline

Допустимый вариант для первой версии:

- синхронная обработка просроченных сессий перед обработкой нового сообщения

Но предпочтительнее сделать отдельный service-слой, чтобы потом было легко вынести это в фоновый job.

## Сервисы

### `MemoryService`

Отвечает за:

- поиск или создание активной сессии
- сохранение сообщений
- получение последних `N` реплик активной сессии
- получение `personal memory`
- получение последних `K` summaries
- сбор memory-блока под бюджет `C`
- закрытие просроченных сессий
- предоставление полного текстового набора сообщений завершенной сессии для суммаризации без обрезки до `N`

### `SummarizationService`

Отвечает за:

- построение полного текстового transcript завершенной сессии из `summary_text`
- вызов OpenAI structured output
- сохранение `dialog_summary`
- merge `personal memory`
- перевод сессии в `summarized`

### `MemoryRepository`

Отвечает за доступ к SQLite:

- CRUD для users
- CRUD для sessions
- CRUD для messages
- чтение и запись personal memory
- чтение и запись session summaries

## Изменения в текущем коде

### Текущее ограничение

Сейчас `RecentMessageStore` хранит память только в RAM.

Это нужно заменить.

### Что должно измениться

- `RecentMessageStore` заменить на persistent memory layer
- всю работу с памятью вынести из `TelegramBotApp` в `MemoryService`
- `TelegramBotApp` должен только:
  - принять сообщение
  - превратить его в сообщение доменной модели
  - вызвать `MemoryService`
  - получить собранный context
  - вызвать модель
  - сохранить ответ

## Логирование

Логирование должно быть прозрачным.

Обязательно логировать:

- создание новой сессии
- продолжение существующей сессии
- закрытие сессии по timeout
- запуск суммаризации
- успешную суммаризацию
- merge personal memory
- число включенных summaries
- итоговый размер memory-блока относительно `C`
- причины, по которым часть summaries не попала в контекст

Нельзя логировать:

- API keys
- access tokens
- содержимое секретов
- полный чувствительный пользовательский контент без необходимости

## Тестовые сценарии

Обязательные тесты:

- память сохраняется между перезапусками процесса
- у разных пользователей независимая память
- новая сессия создается после неактивности больше часа
- суммаризация запускается по завершенной сессии
- суммаризация строится по всей текстовой части диалога
- суммаризация не обрезается до `N` последних реплик
- в суммаризацию попадают текстовые описания вложений, а не сами вложения
- в активный контекст попадают только последние `N` реплик
- в memory-блок всегда пытается попасть `personal memory`
- summaries подаются не больше `K`
- memory-блок не превышает `C`
- personal memory корректно мержится после нескольких сессий
- повторная суммаризация одной и той же сессии не происходит

## Принятые правила

- память разделяется по `telegram_user_id`
- только одна активная сессия на пользователя
- после часа тишины сессия считается завершенной
- суммаризация идет по всему диалогу, а не по `N` последним репликам
- сами документы повторно в суммаризацию не отправляются
- в новый ответ попадают только `N` последних реплик активной сессии
- из прошлых диалогов в prompt попадают только `personal` и до `K` последних summaries
- итоговый memory-блок ограничен бюджетом `C`
